1 O Problema

O sistema precisava enviar e-mails para gestores de unidades de saúde que não possuíam profissionais cadastrados no CNES. Três abordagens foram tentadas:

Abordagem Problema
Brevo + Gmail FUNCIONAVA PARCIAL A API aceitava os e-mails, mas iam para a caixa de SPAM dos destinatários por usar Gmail como remetente
Brevo + Domínio do Governo BLOQUEADO Exigia autenticação de domínio com registros DNS (SPF, DKIM, DMARC) — sem acesso ao DNS
SMTP Direto BLOQUEADO Portas 25, 587 e 465 do servidor do governo bloqueadas externamente

2 A Descoberta: Zimbra

Ao acessar a URL do webmail informada pelo usuário, descobriu-se que o sistema de e-mail da prefeitura era o Zimbra — uma plataforma de código aberto amplamente utilizada em governos brasileiros.

🔍 Informações obtidas via DNS:

A página de login do Zimbra estava acessível na porta 443 (HTTPS) — mesma porta usada para navegar na internet, nunca bloqueada em firewalls corporativos.

3 A Solução: API SOAP do Zimbra

O Zimbra expõe uma API SOAP completa que permite enviar e-mails programaticamente via HTTPS. O fluxo é simples:

Frontend React
→
Python (Vercel)
→
Zimbra SOAP API
→
📬 E-mail Enviado!
Se o Zimbra falhar, o sistema cai no fallback:
⚠️ Zimbra offline →
Python → Brevo API
→
📨 E-mail via Gmail!

Como funciona a API SOAP

São necessárias duas chamadas:

1. Autenticação — AuthRequest

Envia e-mail e senha para o Zimbra. O servidor valida e retorna um token de autenticação (authToken).

POST /service/soap/
Content-Type: text/xml

<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <AuthRequest xmlns="urn:zimbraAccount">
      <account by="name">gecav.semusa@portovelho.ro.gov.br</account>
      <password>********</password>
    </AuthRequest>
  </soap:Body>
</soap:Envelope>

2. Envio — SendMsgRequest

Usa o token para enviar o e-mail com HTML + texto alternativo.

<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Header>
    <context xmlns="urn:zimbra">
      <authToken>TOKEN_AQUI</authToken>
    </context>
  </soap:Header>
  <soap:Body>
    <SendMsgRequest xmlns="urn:zimbraMail">
      <m>
        <e t="t" a="destinatario@email.com"/>
        <su>[SEMUSA] Cadastro CNES...</su>
        <mp ct="multipart/alternative">
          <mp ct="text/plain">
            <content>Prezado(a)...</content>
          </mp>
          <mp ct="text/html">
            <content><html>...</html></content>
          </mp>
        </mp>
      </m>
    </SendMsgRequest>
  </soap:Body>
</soap:Envelope>
💡 Por que SOAP e não REST? O Zimbra foi criado antes do REST se popularizar. Sua API nativa é SOAP, que apesar de mais verbosa (XML), é extremamente estável e bem documentada.

4 Arquitetura do Sistema

O Vercel, onde o frontend React já estava hospedado, suporta Python Serverless Functions — funções que rodam sob demanda na nuvem. Basta criar um arquivo .py na pasta api/ que o Vercel automaticamente o expõe como endpoint HTTP.

Estrutura de arquivos

monitoramento-cnes/
├── api/
│   └── send-email.py        ← Função Python (Zimbra SOAP)
├── requirements.txt          ← Vazio (smtplib + xml são built-in)
├── vercel.json               ← Config atualizada (Python + Vite)
└── src/components/
    └── UnidadesSemCadastroModal.jsx  ← Frontend com fallback

Fluxo de Dados

Quando o usuário clica em "Enviar E-mail" no modal:

  1. Frontend (React) faz fetch('/api/send-email', { body: dados })
  2. Vercel recebe a requisição e executa api/send-email.py
  3. Python autentica no Zimbra via SOAP (e-mail + senha)
  4. Python envia o e-mail via SendMsgRequest
  5. Vercel retorna JSON { success: true } para o frontend
  6. Frontend exibe toast verde de confirmação

Se a etapa 3 falhar (Zimbra offline), o sistema tenta automaticamente o Brevo (Gmail) como fallback!

No arquivo UnidadesSemCadastroModal.jsx, o envio ocorre em duas etapas:

async function enviarComFallback(payload) {
  // 1. Tenta Zimbra primeiro
  try {
    return await tentarEnviarZimbra(payload);
  } catch (e) {
    // 2. Se falhar, usa Brevo (Gmail)
    return await tentarEnviarBrevo(payload);
  }
}

6 Análise do Código

Principais decisões técnicas tomadas durante a implementação:

DecisãoMotivo
SOAP 1.1 (http://schemas.xmlsoap.org/soap/envelope/) Zimbras mais antigos (2005-2021) preferem SOAP 1.1. SOAP 1.2 poderia falhar silenciosamente
XML via ElementTree (built-in) Evita dependências externas no requirements.txt. Toda máquina Python tem ElementTree
Sem escape manual de HTML O ElementTree.text já escapa automaticamente <, > e & para XML. Escape manual causaria dupla-escapagem (&amp;)
multipart/alternative com texto plano + HTML Garante que e-mail seja legível em qualquer cliente (texto puro para clients básicos, HTML para modernos)
Timeout de 30s Limite seguro para funções serverless sem estourar o tempo máximo do Vercel
Auth token SEM cache Autentica a cada requisição. Cache poderia expirar e causar erros silenciosos
🔧 Detalhe importante: O `Content-Type` usado é text/xml (padrão SOAP 1.1), não application/soap+xml (SOAP 1.2). Isso garante compatibilidade com versões mais antigas do Zimbra.

7 Deploy e Configuração

Comandos executados

# 1. Link do projeto ao Vercel
vercel link --project monitoramento-cnes

# 2. Configurar variáveis de ambiente
vercel env add ZIMBRA_URL production
vercel env add ZIMBRA_USERNAME production
vercel env add ZIMBRA_PASSWORD production   ← criptografada
vercel env add FROM_EMAIL production
vercel env add FROM_NAME production

# 3. Deploy para produção
vercel deploy --prod --yes
